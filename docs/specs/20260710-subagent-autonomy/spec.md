# 子代理自主派工授權

> 狀態：設計已核准，待實作

## 問題

Codex 的保守預設要求使用者或適用的 Agent 規則明示允許，才會派出子代理。
本倉目前只要求派工前諮詢 `model-tier-prompting`，沒有明確授予自主派工權，
因此 Agent 可能把「使用者未主動要求子代理」誤當成不派工的理由。

## 目標

1. 明確允許本倉 Agent 自主判斷是否使用子代理，不需要使用者逐次批准。
2. 保留原始任務範圍、權限、外部副作用與破壞性操作的既有批准邊界。
3. 把 `model-tier-prompting` 定位為派工品質指引，而不是派工許可閘。
4. 未來由本倉為其他 subject 導入、同步或升級 harness 時，補齊缺少的等價授權。

## 本次範圍

- 修改本倉根 `AGENTS.md`；`CLAUDE.md` 透過既有 symlink 同步生效。
- 擴充既有可信集檢查，防止授權條款或後續升級規則被意外移除。
- 更新本設計文件中的實作狀態與驗證證據。

本次不修改任何 subject、agent-kit 模板或本機客戶端安裝樹，也不新增 hook、
skill 或強制派工機制。

## 約束層文本

現有「派工前先查層級」章節改為「子代理自主派工」，內容表達以下契約：

> 本倉明確允許並授權 Agent 自主使用子代理。只要工作仍在使用者原始任務範圍內，
> 且 Agent 判斷委派、平行處理或上下文隔離能提升品質、速度或可靠性，就可以直接
> 派出子代理；不需要使用者主動要求或逐次批准。不得把「使用者未明示要求使用
> 子代理」當成不派工的理由。
>
> 這項授權不擴張任務範圍或操作權限。子代理繼承本倉規則及既有 permission /
> approval 邊界；破壞性操作、外部副作用或實質範圍變更，仍須依原規則取得授權。
> 主 Agent 對任務拆分、結果整合與最終驗證負責，不得把子代理的完成聲明直接當成
> 驗證證據。
>
> 決定派工後，先諮詢 `model-tier-prompting` 做層級自評：依 roster 排序當前可用
> 模型，再按任務難度選擇子代理形態、模型與提示厚度。這是派工品質指引，不是使用
> 子代理的許可閘。

這是允許自主判斷，不是要求每個任務都必須派工。緊密耦合、無法獨立驗收，或派工
成本高於收益的工作，可以由主 Agent 直接完成。

## 後續 subject 升級規則

使用本倉為其他 subject 導入、同步或升級 harness 時，Agent 必須檢查目標倉實際
生效的 Agent 規則：

1. 已有語義等價的自主派工授權：不重複添加。
2. 沒有等價授權：把本節的授權與安全邊界納入該次升級。
3. 已有明確且更嚴格的限制：不靜默覆寫；保留限制並向使用者報告差異。

本次只把這項要求寫入本倉規則，不立即修改任何 subject，也不以字串 patch 自動改寫
不同倉庫的 Agent 文件。由執行升級的 Agent 做語義判斷，避免重複條款或破壞目標倉
既有規範。

## 強制與驗證

現有 `tools/harness/checks.py` 已檢查根 `AGENTS.md` / `CLAUDE.md` 的派工層級指引。
實作時先補 Red 測試或檢查條件，再擴充這個契約，使可信集至少驗證：

- 存在「子代理自主派工」章節。
- 明確表達不需要使用者主動要求或逐次批准。
- 明確保留權限與任務範圍邊界。
- 明確要求未來升級缺少等價授權的 subject 時補齊。
- 保留 `model-tier-prompting` 指針，且把它定位為 advisory 而非許可閘。

實作完成後運行 `bash tools/harness/test-harness.sh` 作為全量可信集；只跑窄測不能
宣稱收環。

## 驗收標準

1. Agent 可從根規則直接判定本倉已授權自主派工，無需額外向使用者索取派工許可。
2. 規則不會被解讀為強制派工，也不會擴張任務或操作權限。
3. 主 Agent 保留拆分、整合與驗證責任。
4. 未來 subject 升級有明確的「已有則跳過、缺少則補齊、衝突則保留並報告」流程。
5. 本次 git diff 不包含任何 subject、模板或客戶端安裝樹變更。
6. 全量可信集通過。
